## 1. Jinja2 템플릿 도입

### 개념
FastAPI의 `Jinja2Templates`를 활용해서 HTML을 분리 (`templates/` 디렉터리).

### 장점
- 뷰/로직 분리 (MVC-ish)
- HTML 수정 시 Python 코드 변경 불필요
- 재사용, 상속, 번들링, i18n/프론트엔드 협업이 쉬움
- 코드가 더 깔끔하고 테스트 대상 축소

### 단점
- 초기 구조 변경 비용
- 템플릿 파일 추가 관리 필요
- 작은 학습 비용 (템플릿 문법, `templates.TemplateResponse`)
- API 전용 서비스면 HTML 분리는 불필요할 수 있음

### 예시
- `app.get("/list")` → `return templates.TemplateResponse("list.html", {"request": request, "tils": tils})`

## 2. Repository 패턴 적용

### 개념
데이터 접근을 추상화 (`TilRepository` 인터페이스) + 구현체 (`InMemoryTilRepository`, `SqliteTilRepository` 등)

### 장점
- 전역 상태 제거, DI 가능
- 교체 가능한 저장소 (테스트용 인메모리와 실제 DB 분리)
- 비즈니스 로직(`서비스`)과 CRUD 분리하여 SRP 준수
- 테스트가 독립적이고 명확해짐

### 단점
- 코드 양 증가 (추상/구현/서비스)
- 단순 앱에선 오버엔지니어링 가능
- 테스트/디자인 추가 필요
- 의존주입/팩토리 거의 필수 → 초기 구조 부담

### 예시
- `class TilRepository: async def list()`, `async def add(til)`
- `app = FastAPI(dependencies=[Depends(get_repository)])`

## 3. Pydantic 모델 도입

### 개념
`TilIn`, `TilOut` 같은 `BaseModel` 정의 + 입력 검증, 직렬화/문서화 일원화.

### 장점
- 입력/출력 구조, 필수/옵션 자동 검증
- `title`/`content` 타입, 길이, 공백 체크 등 선언적 제한
- 코드 자가 문서화 (`OpenAPI`) 및 IDE 자동완성
- 엔티티/DTO 명확화로 코드 가독성 향상

### 단점
- 간단 앱에선 약간의 오버헤드
- `dict`에서 모델 변환, 데이터 마샬링 작업 필요
- 런타임 타입 변환 확인(별도 시나리오)
- SQLAlchemy 등 ORM과 통합 시 추가 매핑 필요

### 예시
- `class TilCreate(BaseModel): title: constr(min_length=1); content: constr(min_length=1)`
- `@app.post("/submit") async def submit_til(item: TilCreate)`
